# AGENTS.md

## 1. 프로젝트 목적

이 프로젝트는 쿼카를 중심으로 행복, 힐링, 생태 보호의 메시지를 전달하는 정적 브랜드 웹사이트다.

코드를 수정하는 에이전트는 단순히 화면을 추가하는 데 그치지 않고 다음 기준을 함께 지켜야 한다.

- 기존의 따뜻하고 귀여운 브랜드 분위기 유지
- 초보자도 읽을 수 있는 단순한 코드 작성
- 데스크톱·태블릿·모바일 반응형 지원
- 접근성과 기본 웹 표준 준수
- 외부 프레임워크 없이 Vanilla HTML/CSS/JavaScript 사용

---

## 2. 기술 스택

허용되는 기본 기술:

- HTML5
- CSS3
- Vanilla JavaScript
- Google Fonts
- 브라우저 기본 Web API

현재 사용 중인 Web API:

- `navigator.share`
- `navigator.clipboard`
- `window.location`

다음 도구는 명시적인 요청이 없으면 추가하지 않는다.

- React
- Vue
- Angular
- Bootstrap
- Tailwind CSS
- jQuery
- npm 패키지
- 번들러 및 빌드 도구

## 3. 현재 페이지의 주요 구조

```text
body
├─ header.header
│  └─ nav.nav
│     ├─ logo
│     ├─ mobile menu button
│     └─ navigation links
├─ main
│  ├─ hero
│  ├─ intro
│  ├─ charm cards
│  └─ protect CTA
├─ footer
└─ toast
```

주요 ID:

| ID | 역할 |
|---|---|
| `menuBtn` | 모바일 메뉴 열기·닫기 |
| `navLinks` | 모바일 메뉴 영역 |
| `shareBtn` | 공유 기능 실행 |
| `campaignBtn` | 캠페인 안내 |
| `goodsBtn` | `goods.html` 이동 |
| `toast` | 짧은 상태 메시지 표시 |
| `intro` | 소개 섹션 앵커 |
| `charm` | 매력 섹션 앵커 |
| `protect` | CTA 섹션 앵커 |

기존 ID나 클래스 이름을 변경할 때는 HTML, CSS, JavaScript의 모든 참조를 함께 수정한다.

## 4. 수정 원칙

### 반드시 지킬 것

1. 기존 기능을 먼저 파악하고 수정한다.
2. 요청과 관련 없는 코드까지 대규모로 재작성하지 않는다.
3. 한 기능에는 가능한 한 하나의 명확한 책임을 부여한다.
4. 새 요소는 기존 디자인 토큰과 컴포넌트를 재사용한다.
5. 반응형과 접근성 검토 없이 작업을 완료했다고 판단하지 않는다.
6. 실제 연결되지 않은 버튼은 작동하는 것처럼 위장하지 않는다.
7. 사용자에게 보여주는 문구는 자연스러운 한국어로 작성한다.

### 피해야 할 것

- 같은 색상 값을 여러 곳에 하드코딩하기
- 불필요한 전역 변수 추가
- 인라인 `onclick` 이벤트 사용
- 의미 없는 `div`만 반복해서 사용
- 버튼을 링크처럼 사용하거나 링크를 버튼처럼 사용하기
- 모바일 레이아웃을 데스크톱 작업 후 임시로 덧붙이기
- 기존 `prefers-reduced-motion` 처리를 제거하기
- 외부 이미지의 이용 가능 여부를 확인하지 않고 무분별하게 추가하기

## 5. HTML 규칙

### 시맨틱 요소

가능한 경우 다음 요소를 우선한다.

- 페이지 머리말: `<header>`
- 내비게이션: `<nav>`
- 주요 내용: `<main>`
- 독립 콘텐츠 묶음: `<section>`
- 카드형 독립 콘텐츠: `<article>`
- 페이지 하단: `<footer>`
- 이동: `<a>`
- 동작: `<button>`

### 링크와 버튼 구분

다른 페이지 또는 위치로 이동하면 `<a>`를 사용한다.

```html
<a href="./goods.html" class="btn btn-outline">굿즈 아이디어 보기</a>
```

현재처럼 JavaScript 이동을 사용할 수 있지만 단순 페이지 이동이라면 링크가 더 적합하다.

사용자 행동을 실행하면 `<button>`을 사용한다.

```html
<button type="button" class="btn btn-soft" id="shareBtn">
  행복 공유하기
</button>
```

### 접근성

- 모든 `<img>`에 `alt`를 작성한다.
- 장식 목적이면 `alt=""`를 사용한다.
- 아이콘만 있는 버튼은 `aria-label`을 작성한다.
- 토글 버튼은 `aria-expanded` 상태를 동기화한다.
- 새 섹션의 `id`는 중복되지 않아야 한다.

## 6. CSS 규칙

### 디자인 토큰 우선

색상, 그림자, 반지름, 컨테이너 크기는 `:root` 변수를 사용한다.

```css
:root {
  --yellow: #FFD86B;
  --brown: #A9784B;
  --cream: #FFF7E8;
}
```

새로운 반복 값이 생기면 변수를 추가한다. 한 번만 쓰는 값까지 과도하게 변수화할 필요는 없다.

### 클래스 이름

- 소문자 kebab-case를 사용한다.
- 역할을 나타내는 이름을 사용한다.
- 시각적 위치만 나타내는 이름은 피한다.

좋은 예:

```css
.goods-card
.goods-card-title
.campaign-banner
```

피할 예:

```css
.left-box
.big-yellow-thing
.box2
```

### 컴포넌트 재사용

새 버튼을 만들 때 `.btn`의 공통 규칙을 재사용한다.

```html
<a class="btn btn-primary" href="#">...</a>
```

카드, 섹션 제목, 페이지 컨테이너도 기존 구조를 우선 재사용한다.

### 반응형

현재 기준점:

```css
@media (max-width: 1024px) { }
@media (max-width: 700px) { }
@media (max-width: 430px) { }
```

새 breakpoint는 꼭 필요한 경우에만 추가한다. 먼저 기존 기준점 안에서 해결 가능한지 검토한다.

### 상태 스타일

새 인터랙티브 요소에는 다음 상태를 검토한다.

- 기본
- Hover
- Focus-visible
- Active
- Disabled

권장 Focus 스타일:

```css
:focus-visible {
  outline: 3px solid var(--yellow);
  outline-offset: 3px;
}
```

### 모션

모든 새 애니메이션은 다음 설정에서 제거되도록 작성한다.

```css
@media (prefers-reduced-motion: reduce) {
  /* animation과 transition 제거 */
}
```

## 7. JavaScript 규칙

### 기본 스타일

- `const`를 우선 사용하고 재할당이 필요할 때만 `let`을 사용한다.
- DOM 요소는 파일 상단에서 목적별로 모아 선택한다.
- 이벤트는 `addEventListener`로 연결한다.
- 함수 이름은 동작이 드러나게 작성한다.
- 기능이 커지면 이벤트 콜백 내부 코드를 별도 함수로 분리한다.

좋은 예:

```javascript
function openGoodsPage() {
  window.location.href = "./goods.html";
}

goodsBtn.addEventListener("click", openGoodsPage);
```

### DOM 선택 시 안전성

여러 페이지가 같은 JavaScript를 공유하게 되면 요소가 존재하는지 검사한다.

```javascript
if (goodsBtn) {
  goodsBtn.addEventListener("click", openGoodsPage);
}
```

현재처럼 한 HTML 안에 스크립트가 있고 대상 요소가 항상 존재하면 필수는 아니지만, 공통 스크립트로 분리할 때 반드시 적용한다.

### 오류 처리

- 브라우저 API는 지원 여부와 실패 가능성을 고려한다.
- 사용자에게 필요한 결과는 Toast 또는 화면 메시지로 알린다.
- 개발자용 오류와 사용자용 메시지를 구분한다.

```javascript
try {
  await navigator.clipboard.writeText(text);
  showToast("공유 문구가 복사됐어요 ☺");
} catch (error) {
  console.error(error);
  showToast("복사하지 못했어요. 다시 시도해 주세요.");
}
```

### 페이지 이동

단순 이동은 가능하면 HTML 링크로 처리한다. JavaScript가 필요한 이동만 `window.location.href`를 사용한다.

## 8. 콘텐츠 규칙

### 톤

- 따뜻하고 친근하게 작성한다.
- 지나치게 유아적이거나 가볍게만 보이지 않도록 한다.
- 보호 메시지는 죄책감을 강요하지 않는다.
- 한 문장에 너무 많은 정보를 넣지 않는다.

### 사실성

쿼카의 생태, 분포, 보호 상태, 행동을 새로 작성할 때는 신뢰할 수 있는 자료로 확인한다.

검증되지 않은 표현 예:

- “쿼카는 항상 웃는다.”
- “쿼카는 사람을 매우 좋아한다.”
- “쿼카는 전 세계 어디서나 볼 수 있다.”

브랜드 표현과 생물학적 사실을 구분한다.

- 브랜드 표현: “웃는 듯한 얼굴”
- 피해야 할 단정: “행복해서 웃는 얼굴”

## 9. 이미지와 외부 리소스

현재 페이지는 다음 외부 리소스에 의존한다.

- Google Fonts
- 외부 URL의 쿼카 이미지

운영용 배포에서는 다음을 권장한다.

1. 사용 권한을 확인한 이미지를 프로젝트 에셋 폴더에 저장한다.
2. 파일명을 의미 있게 작성한다.
3. 이미지 크기를 실제 표시 크기에 맞게 최적화한다.
4. WebP를 우선 고려한다.
5. 외부 URL 장애로 화면이 깨지지 않도록 한다.

예시 구조:

```text
assets/
├─ images/
│  ├─ quokka-hero.webp
│  └─ goods/
├─ icons/
└─ styles/
```

단일 HTML 유지가 요구된 경우에는 구조 분리를 강제로 진행하지 않는다.

## 10. 굿즈 페이지 작업 지침

현재 메인 페이지의 `goodsBtn`은 `./goods.html`로 연결된다.

`goods.html`을 구현하거나 수정할 때 다음을 유지한다.

- 메인 페이지와 같은 헤더 및 로고
- 같은 색상과 폰트
- 같은 `.page` 컨테이너 너비
- 카드형 상품 그리드
- 모바일 1열, 태블릿 2열, 데스크톱 다열 구조
- 메인 페이지로 돌아가는 명확한 링크

예상 굿즈 항목:

- 쿼카 소개 카드
- 인형
- 티셔츠
- 후드
- 안아주기 베개
- 향기 봉지
- 슬리퍼
- 안대

상품을 실제 판매하지 않는 기획 페이지라면 가격, 구매 가능 여부, 재고를 사실처럼 표시하지 않는다.

## 11. 작업 절차

에이전트는 변경 작업을 다음 순서로 수행한다.

1. 요청 사항과 기존 코드의 관련 부분을 찾는다.
2. 기존 클래스·토큰·함수를 재사용할 수 있는지 확인한다.
3. 가장 작은 범위로 코드를 수정한다.
4. HTML 문법과 태그 닫힘을 확인한다.
5. JavaScript 콘솔 오류를 확인한다.
6. 아래 화면 너비에서 레이아웃을 확인한다.
   - 1440px
   - 1024px
   - 768px
   - 430px
   - 360px
7. 키보드 Tab 이동과 Focus 표시를 확인한다.
8. 메뉴, 공유, Toast, 페이지 이동 기능을 확인한다.
9. 변경 사항을 짧게 설명한다.

## 12. 완료 조건

다음 조건을 모두 충족해야 작업 완료로 판단한다.

- [ ] HTML에 명백한 문법 오류가 없다.
- [ ] 브라우저 콘솔에 새 오류가 없다.
- [ ] 기존 메뉴와 버튼 기능이 깨지지 않았다.
- [ ] 변경된 UI가 기존 컬러와 타이포그래피에 어울린다.
- [ ] 데스크톱과 모바일에서 가로 스크롤이 없다.
- [ ] 이미지에 적절한 대체 텍스트가 있다.
- [ ] 키보드로 주요 기능을 사용할 수 있다.
- [ ] `prefers-reduced-motion` 설정이 유지된다.
- [ ] 새 콘텐츠의 생태 관련 사실을 검증했다.
- [ ] 요청하지 않은 프레임워크나 의존성을 추가하지 않았다.

## 13. 응답 형식

코드 변경 후 사용자에게는 다음 내용을 간단히 보고한다.

1. 무엇을 변경했는지
2. 어떤 파일을 변경했는지
3. 사용자가 직접 확인해야 할 부분이 있는지
4. 알려진 제한이나 임시 동작이 있는지

초보 사용자가 이해할 수 있도록 어려운 용어만 나열하지 말고, 코드가 왜 그렇게 동작하는지도 짧게 설명한다.
